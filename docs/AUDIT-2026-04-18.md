# Claude Budget Dispatcher — Hardening Audit

**Date:** 2026-04-18
**Scope:** All six sections of `docs/DEEP-RESEARCH-PROMPT.md` (Security, Reliability, Performance, Dashboard UX, Operations, Multi-Machine)
**Methodology:** Two parallel read-only code audits (Sections A+B and C–F), followed by a manual verification pass. Every finding cites a specific file and line number.
**Files examined:** `package.json`, `package-lock.json`, `config/budget.json`, all 23 `.mjs` files under `scripts/` and `scripts/lib/`, plus `scripts/tray-app.cs` and the status directory.

---

## Executive summary

The dispatcher is well-architected and defensive-in-depth on the items you've explicitly thought about (path traversal, git push blocking, subprocess env filtering on the test runner, H1/H3 ceremonies, JSONL lenient parsing, security scanner before commit). Most threat vectors you asked about are already closed. The real weaknesses are operational and fleet-level, not code-safety-level:

- **No distributed lock** — the status board is advisory; two machines can dispatch the same project at the same time.
- **No out-of-band alerting** — when health goes `down`, nothing tells you unless you look at the tray or the dashboard.
- **Non-router modules are untested** — `worker.mjs`, `gates.mjs`, `provider.mjs`, `estimate-usage.mjs`, `schemas.mjs`, `health.mjs` have zero unit tests covering the functions with highest blast radius.
- **Dashboard dispatch/pause endpoints are localhost-bound but unauthenticated** — any page you visit in the same browser session can POST to `http://127.0.0.1:7380/api/dispatch` via DNS rebinding.
- **Subprocesses other than `runTestsSafe` inherit full env** — `estimate-usage.mjs`, `npm audit`, and PowerShell invocations see all `*_API_KEY` variables.
- **`rotateLog()` exists but is never called** — one-line fix that currently isn't wired in.

Nothing found in this audit is an immediate code-execution or credential-compromise hole in the strict sense. The corrections below are hardening, not emergency response.

### ⚠ Correction to an earlier audit pass

The Section A/B audit agent initially flagged `dashboard.mjs` as CRITICAL because it claimed the server binds to all network interfaces. **That finding was wrong.** Line 694 of `dashboard.mjs` is explicitly:

```js
server.listen(PORT, "127.0.0.1", () => { ... });
```

The server is correctly bound to loopback. The underlying concern — unauthenticated POST mutation endpoints with no Host-header check or CSRF protection — is still real (DNS rebinding by a malicious webpage can hit them), but that's a **HIGH**, not a **CRITICAL**. This is captured as finding **A5.1** below.

## A. Security & Trust Boundaries

### A1. Prompt injection surface area — MEDIUM

**Where untrusted content enters LLM prompts:**

| Injection point | File : line | Content source |
|---|---|---|
| Selector prompt (project + task selection) | `scripts/lib/selector.mjs:139–184` | STATE.md + DISPATCH.md excerpts from every project |
| Pre-approved section extraction | `scripts/lib/context.mjs:39–44, 39–81` | `DISPATCH.md` parsed by regex, not escaped |
| Codegen / docs / analysis prompts | `scripts/lib/worker.mjs:852–937` | Whole source files via `gatherSrcFiles()`, wrapped in triple-backtick fences |
| Audit prompt | `scripts/lib/worker.mjs:807–814` | Changed files embedded in a JSON-format audit request |

**What's already protecting you:**

- Selector returns a strict JSON schema validated post-call against the rotation list (`selector.mjs:112–124`) — injected content can't select a project that isn't in `budget.json`.
- Worker-generated code is not executed in the dispatcher process. It's written to a worktree, tests run in a subprocess with `SAFE_ENV_KEYS`, commits stay on `auto/*` branches that cannot push.
- File content is truncated to 50 KB per file (`MAX_FILE_CHARS` in `worker.mjs:12`).
- The security scanner (`scan.mjs:13–27`) blocks commits that contain API key patterns.

**What's missing:**

- No XML/markdown escaping on the untrusted content. A malicious `DISPATCH.md` can include `</data>` closing tags, fake `## Pre-Approved Tasks` headers, or prompt-style directives that confuse the LLM's reasoning about *which* project+task to pick. Post-call validation catches "picked nonexistent project," but it doesn't catch "picked a legitimate-but-wrong project because the attacker tricked the LLM's ranking."
- Cross-project scope: selector sees all 10 projects in one prompt. Project A's DISPATCH.md can influence which of the *other* projects is selected next.

**Fix (Backlog):** Wrap untrusted content in a sentinel like `<untrusted-project-content project="burn-wizard">…</untrusted-project-content>` and add a system-prompt instruction to treat the block as data, not directives. Optionally, hash-verify DISPATCH.md against a known-good version stored in `status/`.

### A2. Credential exposure — HIGH

**Well-handled:** `worker.mjs:24–38` defines a `SAFE_ENV_KEYS` allowlist (~20 env vars, no `*_API_KEY`). It is applied in three places:

- `executeLocalTask()` — `worker.mjs:169`
- `runTestsSafe()` — `worker.mjs:682`
- `verify-commit.mjs:170`

**Gap — subprocesses that inherit full env:**

| Call | File : line | Inherits |
|---|---|---|
| `execFileSync("node", [estimatorScript])` | `scripts/lib/gates.mjs:43` | Full `process.env` including `GEMINI_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`, `OPENROUTER_API_KEY` |
| PowerShell invocation for scheduler query | `scripts/dashboard.mjs:88` | Full env |
| `git fsck`, `npm audit` | `scripts/lib/git-lock.mjs:75, 204` | Full env |

**Risk scenario:** A compromised transitive npm dependency with a preinstall or postinstall hook runs during the next `npm audit` or fresh `npm ci` and exfiltrates the API keys via DNS or out-of-band HTTP. This is general npm supply-chain risk, but the dispatcher's weekly `npm audit` trigger amplifies the window.

**Fix (Do now):** Export `SAFE_ENV_KEYS` + `getSafeTestEnv()` from `worker.mjs` (or move to a shared `env.mjs`) and apply `env: getSafeTestEnv()` to every `execFileSync` / `execSync` / `spawn` call outside of provider/worker flows. The estimator genuinely has no need for API keys.

### A3. Path traversal — INFO (design is sound)

`isPathInside()` at `scripts/lib/worker.mjs:54–90` combines:

1. `realpathSync()` ancestor walk to canonicalize even not-yet-existing paths.
2. Windows reserved-name regex (`/^(?:con|prn|aux|nul|com[1-9]|lpt[1-9])(?:$|\.)/i`) — correctly rejects `CON.txt`, `COM1`, etc.
3. Separator-aware prefix comparison with case-insensitive match on Windows.

It's called twice on the codegen path (`worker.mjs:269–278, 634`) before any write happens. Attempted escapes verified mentally against the code:

- `src/evil → /etc/passwd` symlink: realpath resolves to `/etc/passwd`, prefix check fails, write rejected.
- Absolute path in LLM output (`/etc/passwd`, `C:\Windows\System32\drivers\etc\hosts`): resolved outside project, prefix check fails.
- `..`, `..\`, URL-encoded variants: path resolution normalizes them; prefix check fails.
- `CON`, `CON.txt`, `COM9`: caught by `WIN_RESERVED`.

**Residual TOCTOU:** The ancestor walk has a race window where a parent directory could become a symlink between `realpathSync()` and the write. In practice the worktree is dispatcher-owned and the window is sub-millisecond, so this is acceptable. Worth a code comment but not a fix.

### A5. Network exposure — HIGH (corrected from earlier CRITICAL false-positive)

**Bind address — verified safe:** `scripts/dashboard.mjs:694` reads `server.listen(PORT, "127.0.0.1", ...)`. The dashboard is loopback-only; remote network access is not possible.

**Real issues on mutation endpoints (`dashboard.mjs:680–688`):**

```
POST /api/engine             → setEngineOverride
POST /api/pause              → togglePause
POST /api/dry-run            → setDryRun
POST /api/dispatch           → triggerDispatch
POST /api/projects/reorder   → reorderProject
POST /api/projects/tasks     → updateProjectTasks
```

None of these check:

- **`Host` header** — a malicious page on `attacker.com` can use DNS rebinding to point `attacker.com` at `127.0.0.1`, then your browser's same-origin check still passes because the page's origin *is* `attacker.com`. The browser will happily POST to the rebound address.
- **`Origin` / `Referer` header**.
- **CSRF token** — any authenticated web session where the dashboard is running is vulnerable.
- **CORS** — no `Access-Control-Allow-Origin` header is set, which means simple-request POSTs (`Content-Type: application/json` disqualifies simple-request, so this helps slightly, but POST with `text/plain` or form-encoded is still possible).

**Impact if exploited:** Attacker can pause the dispatcher, trigger a dry-run dispatch (and read the JSON response, leaking project names/tasks), flip engine override, or reorder project rotation. Not RCE, but it undermines your operational controls.

**Fix (Do now — ~2 hours):**

1. Add a Host-header allowlist at the top of the request handler:
   ```js
   const allowedHosts = new Set([`localhost:${PORT}`, `127.0.0.1:${PORT}`, `[::1]:${PORT}`]);
   if (req.method !== "GET" && !allowedHosts.has(req.headers.host)) {
     res.writeHead(403); res.end("Forbidden"); return;
   }
   ```
2. For defense in depth on mutation endpoints, require a CSRF token. On page load, the dashboard reads `GET /api/csrf-token` (returns a random 32-byte token stored in a module-scope variable valid for the process lifetime); every POST includes it as `X-CSRF-Token`. The allowed-host check above alone already defeats DNS rebinding, so CSRF is belt-and-braces.

### A6. Git safety — INFO (sound)

- `getChangedFiles()` uses `git diff --name-only` + `git ls-files --others --exclude-standard` (`verify-commit.mjs:291–309`), so `.gitignore` is respected — `.env`, `node_modules/`, etc. will never be staged.

No action needed. A one-line code comment next to the H1 ceremony explaining *why* the pushurl sentinel exists would help future readers.

## B. Reliability & Error Handling

### B1. Estimator accuracy — INFO

`estimate-usage.mjs` is mature:

- Streamed parse via `createReadStream` + `readline` — memory-safe for large transcripts.
- **H3 hot-file guard (`:77–104`):** if a file was modified within the last 30 s, the last line is dropped (may be a partial write from an active Claude session).
- `parseUsageLine` wraps `JSON.parse` in try/catch; corrupt lines are silently skipped (`:106–123`).
- Per-file errors logged but non-fatal (`:483–491`).
- `Date.parse` + `Number.isNaN` guard on timestamps.

One documented assumption worth a comment: if Claude Code keeps a transcript open for more than 30 s between writes, the estimator will drop the last line on every run — underestimating budget slightly. This fails *closed* (we skip dispatches we could have run), which is the correct direction for safety.

---

### B2. Gate ordering & TOCTOU — LOW

Gate chain in `dispatch.mjs:97–112` via `runGates()` in `gates.mjs:21–113`:

1. PAUSED file check
2. Budget gate (runs `estimate-usage.mjs` subprocess — ~1–2 s today, could be 10 s+ at scale)
3. Activity gate (transcript mtime — cheap)
4. Daily quota

**Windows:**

- Activity gate is never rechecked after it passes. A 5–10 minute dispatch started 20 min after your last keystroke can complete just as you come back. Impact: one wasted dispatch cycle plus some CPU/budget noise. Acceptable, because the LLM calls aren't cleanly interruptible anyway.
- Daily quota is checked once. If two machines read the same counter value and both proceed, you can exceed `max_runs_per_day` by one or two. This is one of the things the distributed lock (F1.1) would solve for free.
- Budget gate is soft for the free-tier engines (Gemini/Mistral don't consume Claude Max), so a budget swing mid-dispatch is informational only.

**Fix (Backlog):** log a warning if > 5 min elapsed between gate pass and Phase 2, and re-check activity right before the selector call. Keep daily-quota recount inside the distributed-lock path when F1.1 lands.

---

### B3. Worker crash recovery — MEDIUM

**Already handled:**

- `finally` block in `dispatch.mjs:239–244` restores H1 pushurl on every exit path.
- `sweepStaleIndexLocks()` at `git-lock.mjs:27–48` removes `.git/index.lock` older than 30 min at dispatcher startup.
- `sweepStaleWorktrees()` at `git-lock.mjs:112–180` removes worktrees older than 7 days at startup.

**Gap:** 7 days of grace at 8 dispatches/day × ~500 MB per worktree = ~28 GB disk bloat if worker crashes cluster. The sweep runs at next dispatcher startup, so a dispatcher that won't start (because disk is full) creates a vicious cycle.

**Fix (Next sprint):**

1. Reduce grace period to 1 day.
2. On startup, detect any worktree with uncommitted changes and `git reset --hard` before using it, or delete it outright.
3. Log a warning line for each stale worktree swept, so you notice if crashes cluster.

---

### B4. JSONL log corruption — LOW

`appendLog()` at `log.mjs:22–26` writes `JSON.stringify(record) + "\n"` in one `appendFileSync` call. For records < 64 KB (all dispatch records fit), both POSIX and NTFS guarantee atomicity. Reader side (`readLogLines` in `dashboard.mjs:39–43`) splits on `\n` and filters empties; `parseLogLine` returns null on parse failure and callers skip nulls. A partially-written line at the tail won't crash anything.

**Caveat:** if you ever move the log to a shared network FS (SMB/NFS), atomic guarantees weaken. Today the Windows Task Scheduler mutex (`Global\claude-budget-dispatcher` in `run-dispatcher.ps1`) prevents concurrent dispatchers per-machine, and the fleet architecture keeps logs per-machine, so this is theoretical.

**Fix (Backlog):** code comment documenting the atomic-write assumption; add an explicit `flock`/`LockFileEx` wrapper only if you consolidate logs to a shared path in the future.

---

### B5. Model fallback chain exhaustion — LOW

`callModelWithFallback()` at `worker.mjs:427–450` tries each candidate exactly once; retries only on 429 or 5xx *by moving to the next candidate*, not by retrying the same model. With two candidates and worst-case 503 on both: 2 API calls + 2 throttle waits (~42 s total) before throw.

The selector (`selector.mjs:194–237`) *does* have per-model retry with exponential backoff — asymmetry worth noting.

**Fix (Backlog):** add `callGeminiWithRetry`-style exponential backoff (2s, 4s, 8s) per candidate in `callModelWithFallback`. Track and log API-call count per dispatch so you can see the actual worst case in logs.

---

### B6. Dashboard resilience — LOW

- Malformed `budget.json` → `readJson` returns null, callers handle gracefully.
- Malformed JSONL log → per-line parse errors skipped.
- Module-scope caches (`_taskCache`, `_gistCache` at `dashboard.mjs:67–82, 521–566`) have TTL invalidation; don't grow unbounded.
- Big log files: if `rotateLog` never runs (see E2.1) and log reaches 100 MB+, every API call allocates a ~300 MB array of parsed entries. Dashboard will get slow but won't crash.

**Fix (Backlog, cheap):** in `readLogLines`, `statSync` the log first; if size > 10 MB return an error payload and a message pointing the user at `rotateLog`. Combined with the E2.1 fix, this becomes belt-and-braces.

## C. Performance & Scalability

### C1. Log scanning efficiency — MEDIUM (gets worse over time)

Nine dashboard endpoints each call `readLogLines()` which reads and parses the entire JSONL every invocation: `getAnalytics`, `getBudgetDetail`, `getLogs`, `predict`, `getProjects`, `getFleetData`, `getState`, plus the state/predict/budget auto-refresh loops. Today the log is 1.6 KB (10 entries). At 50 entries/day it's ~1.8 MB after 6 months; the Status tab alone reads it every 30 s = 3.6 MB/min just for Status.

**Fix (Next sprint):**

1. Remove Status tab's dependency on the log — it only needs `health.json` + `last-run.json`.
2. Cache the analytics object for 60 s (server-side), bust on `appendLog`.
3. Long-term: maintain a side-index (simple `status/log-index.json` with `{ date → byte-offset, project → byte-offset[] }`) refreshed incrementally on each `appendLog`.

---

### C2. Dashboard response size & gzip — MEDIUM

- Page template is 93 KB as an inline string (`dashboard.mjs:703–950+`).
- Response handler (`:641–645`) sets `Content-Type` and `Content-Length` only; no `Content-Encoding: gzip`.
- With gzip, ~22 KB — negligible on localhost but matters if you ever tunnel the dashboard to a phone.

**Fix (Backlog):**

- Wrap responses in `zlib.createGzip()` when `accept-encoding: gzip` is present.
- Consider extracting CSS/JS to `scripts/dashboard-assets/*` served statically. Makes diffs smaller and opens the door to caching.

---

### C3. Estimator startup cost — MEDIUM–HIGH (future scaling)

`estimate-usage.mjs:41–58` recursively scans `~/.claude/projects/**/*.jsonl` with no memoization or mtime skip. Today ~50–100 files → 1–2 s. At 500+ files or 1 GB+ total (a year of heavy use), full scan becomes ~10–15 s per dispatch, which blocks the gate.

**Fix (Next sprint):**

1. Persist `lastScanTs` and per-file `(path, mtime, cachedUsageTotal)` in `status/usage-estimate.json`.
2. On next run, skip files whose mtime hasn't changed; re-parse only the tail of hot files.
3. Optionally parallelize with `Promise.all` + bounded concurrency (10 workers).

---

### C4. Provider timeouts — HIGH

All providers share a single `API_TIMEOUT_MS` (imported from `throttle.mjs`, applied in `provider.mjs:78–84, 91–99, 134`). This is a mismatch:

- **Ollama (local):** should be 5–10 s. Currently ~30 s → wasted wait on a hung local process.
- **Gemini/Mistral SDK:** 30 s is fine.
- **Groq:** 30 s is fine (usually sub-second).
- **OpenRouter:** should be 45–60 s — model queuing varies.

Router (`router.mjs:105–111`) doesn't distinguish timeouts from other errors; it just advances to the next candidate. No jitter, no exponential backoff on repeated failures.

**Fix (Do now):**

1. Add per-provider timeouts to `budget.json` (`providers.ollama.timeout_ms`, etc.), default values provided.
2. In `callModelWithFallback`, classify error types: `429 rate-limit`, `5xx transient`, `timeout`, `auth-error`. Retry transients with backoff (1s, 2s, 4s, max 3 tries) before advancing; treat auth errors as immediate non-retryable.
3. Emit a structured log entry for every timeout so you can spot provider degradation in Analytics.

## E. Operational Improvements

### E1. Alerting — HIGH

When `computeHealth()` transitions to `down`, the only surfacing is the red tray icon and red banner in the dashboard. If you're away from the machine, you don't know until you come back.

**Recommendation:** **ntfy.sh** (free, no account) or **Pushover** ($5 one-time, more polished). Both are 2–3 lines to integrate.

```js
// after health state change in health.mjs:
if (prevState !== "down" && state === "down") {
  await fetch("https://ntfy.sh/perry-dispatcher-alerts", {
    method: "POST",
    body: `Dispatcher down on ${os.hostname()}: ${reason}`
  }).catch(() => {});
}
```

**Fix (Do now):** Add an `alerting` block to `budget.json` with `{ enabled, provider, topic_or_token, on_transitions: [] }`, and emit on state changes + weekly "still healthy" heartbeat (so silence ≠ health).

---

### E2. Log retention — LOW

`rotateLog()` at `log.mjs:99–144` is fully implemented — it archives entries older than N days to `status/budget-dispatch-log-archive-YYYY-MM-DD.jsonl`. It's **never called** from `dispatch.mjs`.

**Fix (5 minutes):** add one line to `dispatch.mjs` startup:

```js
import { rotateLog } from "./lib/log.mjs";
rotateLog(7);
```

Per-run logs in `status/dispatcher-runs/*` have no cleanup at all. Add a sweep of files older than 30 days to the same startup block.

### E3. Config validation — MEDIUM

`readJson` is fire-and-forget. Silent-error scenarios that would bite you:

- `"max_runs_per_day": -1` → dispatch proceeds indefinitely (likely — confirm in `gates.mjs`)
- `"reserve_floor_pct": 200` → arithmetic produces negative available budget

## F. Multi-Machine Coordination

### F1. Conflict avoidance — HIGH

`status.mjs` posts comments to a pinned GitHub Issue. This is **advisory**: nothing stops two machines from both entering Phase 2 on the same project at the same time.

**Collision walk-through:**

1. PC reads status board; burn-wizard hasn't run in 27h → proceeds.
2. Laptop reads status board within the same minute; sees the same thing → proceeds.
3. Both create `auto/burn-wizard-<task>-2026-04-18` branches in their local clones. Branch names are identical.
4. Both commit. No push, so no direct conflict — but you get two divergent commits on the same branch name across two machines, plus duplicated Claude Max burn on whichever engine they use.

**Recommended lock (Option 2 in the original prompt): Gist-based mutex.** You already have gist infrastructure via `fleet.mjs`.

```js
// at start of dispatch.mjs, after gates pass
const lock = await readGistFile(gistId, "dispatch-lock.json");
const now = Date.now();
if (lock?.machine && (now - new Date(lock.locked_at).getTime()) < 10 * 60_000) {
  console.log(`[dispatch] lock held by ${lock.machine} since ${lock.locked_at}; skipping`);
  process.exit(0);
}
const lockId = crypto.randomUUID();
await writeGistFile(gistId, "dispatch-lock.json", {
  machine: os.hostname(),
  locked_at: new Date().toISOString(),
  lock_id: lockId
});
// small re-read to confirm we won the race
const confirm = await readGistFile(gistId, "dispatch-lock.json");
if (confirm.lock_id !== lockId) {
  console.log("[dispatch] lost race to another machine; skipping");
  process.exit(0);
}
// ... dispatch ...
// in finally: release lock by writing { machine: null } or deleting the file
```

10-minute TTL auto-releases stale locks if a machine crashes holding one.

**Fix (Do now):** single-day implementation. Covers F1, incidentally makes B2's daily-quota TOCTOU benign.

### F3. Gist sync reliability — MEDIUM

const stale = (m) => (Date.now() - new Date(m.computed_at).getTime()) > 2 * 3_600_000;
