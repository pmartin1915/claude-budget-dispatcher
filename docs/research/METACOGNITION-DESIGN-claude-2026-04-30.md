Architecture Decision Record and Technical Implementation Plan: Metacognition and Autonomous Self-Repair Layer for Decentralized Node.js Fleets
System Context and Vulnerability Analysis
The deployment of a decentralized, autonomous budget dispatcher fleet across diverse Windows-based host machines introduces a highly complex operational matrix. The current technological foundation leverages Node.js utilizing ECMAScript Modules (ESM) for core execution, Git for version control, and the GitHub REST and GraphQL APIs accessed via Octokit for distributed coordination. Security and execution boundaries are presently maintained through Abstract Syntax Tree (AST) mutation validation, execution of untrusted code within an isolated-vm sandbox, and a decentralized coordination mechanism relying on a centralized GitHub Gist containing heartbeat.json, pending-merges.json, and associated locking protocols. Furthermore, a local sentinel.mjs script acts as a basic watchdog, requeuing tasks when it detects that a peer node has failed to update its heartbeat.
Despite these robust mechanisms, the architecture remains vulnerable to two distinct categories of catastrophic failure: in-band cognitive degradation and out-of-band fatal process termination. Autonomous agents, particularly those driven by Large Language Models (LLMs) such as Google GenAI (Gemini) and Mistral, operate within inherently non-deterministic parameters.1 While they excel at single-shot execution, extended multi-step reasoning tasks frequently subject these agents to the "coherence trap".2 In this failure mode, the agent retrieves irrelevant data, generates plausible but incorrect analytical reasoning, and confidently pursues impossible solutions.2 Because the agent lacks internal mechanisms for self-monitoring or objective validation, it fails to recognize the mismatch between its overarching goal and its active execution trajectory.2 This results in infinite logic loops, severe API token depletion, and the continuous generation of degraded code.2
Simultaneously, the physical execution environment presents a secondary vector for systemic collapse. Node.js applications running on Windows architectures are susceptible to unhandled runtime exceptions, memory leaks within the V8 engine, and native module crashes. When such a failure occurs, the operating system abruptly terminates the Node.js process, logging an Application Error categorized under Event ID 1000 in the Windows Event Viewer.4 While the sentinel.mjs script can detect that a node has ceased transmitting its heartbeat and subsequently requeue its tasks to healthy peers, it operates strictly in-band. Once the underlying Node.js process is dead, the sentinel cannot physically restart the local machine, diagnose the memory dump, or modify the source code to resolve the underlying fault.
To achieve true operational autonomy and sustained resilience, the architecture must evolve beyond reactive fault tolerance and distributed task requeuing. The system requires a comprehensive Metacognition and Self-Repair Layer. This layer must introduce a zero-cost semantic drift detection pipeline for internal performance reflection, a highly reliable out-of-band Windows Watcher service for fatal crash detection, and an autonomous LLM-driven self-repair cascade governed by strict algorithmic circuit breakers.
Technology Evaluation for Out-of-Band Fault Detection
The foundation of the self-repair cascade relies on a robust out-of-band supervisor capable of monitoring the Node.js dispatcher process. In the event of a total process failure, this Watcher must accurately detect the termination, capture the fatal stack trace, and orchestrate the repair sequence before initiating a restart. The selection of the appropriate Windows service manager is paramount, as the tool must operate reliably as an elevated background daemon without incurring licensing costs. Several technologies were evaluated against the strict requirements of the decentralized fleet.
The PM2 process manager is widely regarded as an industry standard for managing Node.js applications, offering built-in clustering, log management, and automatic restarts.6 However, PM2 lacks native support for Windows service registration.6 Operating PM2 as a persistent background service on Windows requires a convoluted series of legacy workarounds, such as pm2-windows-startup (unmaintained since 2015), node-windows (which suffers from compatibility issues on modern Windows Server 2022/2025 environments), or custom PowerShell injection scripts.6 Because PM2 inherently runs within the user session space rather than as a core system daemon, it introduces unacceptable fragility to the autonomous recovery pipeline.6
The Non-Sucking Service Manager (NSSM) has historically served as the definitive tool for wrapping arbitrary executables as Windows services.7 NSSM is lightweight and provides fundamental restart throttling to prevent tight CPU loops when an application crashes immediately upon execution.8 It allows administrators to define custom exit actions via the registry, determining whether the service should restart or halt based on specific exit codes.8 However, NSSM suffers from significant modernization deficits. It has not received a stable update in over a decade, lacks a modern graphical interface for rapid local troubleshooting, and fundamentally struggles with managing complex child process trees.10 Crucially, while NSSM can redirect stdout and stderr streams, it lacks built-in log rotation and comprehensive health monitoring integrations.8
The Windows Service Wrapper (WinSW), heavily utilized in Jenkins deployment environments, offers a highly configurable, XML-driven approach to service management.10 While it provides robust execution controls, it is currently in a state of maintenance limbo and lacks the scriptable command-line interface (CLI) fluidity required for automated, decentralized node provisioning.6
Alternative approaches relying entirely on native Windows tools, such as configuring the Windows Task Scheduler to trigger a PowerShell script upon detecting Event ID 1000 (Application Error) via complex XPath XML queries, were also analyzed.4 While this approach requires zero external dependencies, it lacks stateful process management. Task Scheduler cannot synchronously hold a service in a pending state while a repair script executes, making it impossible to establish a reliable, synchronous self-repair pipeline.14
Following a rigorous evaluation, the architectural decision is to mandate the implementation of Servy as the primary out-of-band Watcher daemon.10 Servy is a modern, actively maintained open-source Windows service management tool engineered specifically for enterprise-grade reliability.11 It provides a comprehensive suite of features that perfectly align with the requirements of the self-repair cascade. Servy offers advanced log management, natively redirecting standard output and error streams with built-in size-based and date-based rotation policies, ensuring that the critical crash logs required by the LLM are preserved without exhausting local disk space.11
Most importantly, Servy features an advanced lifecycle management system encompassing Pre-Launch and Post-Launch hooks.11 These hooks support synchronous execution, enabling the Watcher to invoke an external PowerShell repair script when a crash is detected.11 If the Pre-Launch repair script is configured with a strict timeout and exits with an error code, Servy will respect the failure and halt the restart process, effectively quarantining the corrupted node.11 This synchronous, stateful control over the execution pipeline, combined with its dedicated PowerShell module (Servy.psm1) for automated provisioning, renders Servy the optimal, zero-cost Watcher technology for the Windows fleet.11
Architecture Decision Record: Metacognition and Self-Repair
This Architecture Decision Record (ADR) formalizes the structural integration of the Metacognitive Memory, the Local Watcher, and the Global Watchdog components within the existing decentralized topology.
The primary architectural challenge stems from the inherent non-determinism of the LLM agents operating the dispatcher nodes. These agents frequently generate inaccurate outputs or encounter unrecoverable failures, leading to a breakdown of trust and rapid token consumption.1 To address this, the architecture will integrate a secondary, "metacognitive" layer that actively monitors the primary low-code/no-code (LCNC) agent.1 Inspired by human introspection, this layer is designed to predict impending task failures based on a defined set of triggers, such as repetitive actions, semantic divergence, or logic looping.1
The integration of metacognition—defined fundamentally as "thinking about thinking"—enables the AI system to monitor, control, and regulate its own cognitive processes.15 This structural enhancement allows the dispatcher to self-assess its performance trajectory, correct systemic errors in real-time, and adapt to changing codebase environments without requiring human intervention.15
To implement this without incurring prohibitive API costs, the Metacognitive Layer will utilize a purely mathematical, zero-cost pipeline.16 The system will extract the execution logs and pull request rationales from the local budget-dispatch-log.jsonl file. Using a local, CPU-bound embedding model, the Node.js dispatcher will convert these textual logs into semantic vectors. The architecture dictates the use of an Exponential Moving Average (EMA) to maintain a persistent memory of the agent's historical success baseline.16 By calculating the cosine similarity between the current execution vector and the EMA baseline, the system can deterministically quantify semantic drift.16 If this drift exceeds a defined threshold, the metacognitive layer will force the agent to abort its current context window, recognizing that it has fallen into a coherence trap.2
For out-of-band fault detection, the ADR dictates a bifurcated approach, addressing both localized process failures and systemic fleet paralysis. At the local host level, the Servy daemon will act as the primary Watcher.11 Servy will wrap the dispatcher.mjs script as an elevated Windows Service. Upon detecting a fatal crash, Servy will initiate the Autonomous Self-Repair Cascade via its Pre-Launch hook execution mechanism.11 This synchronous hook will invoke a PowerShell orchestrator script designed to extract the fatal stack trace, query a fresh Gemini or Mistral LLM instance for a targeted Abstract Syntax Tree (AST) patch, validate the patch within the local isolated-vm sandbox, and commit the repaired code to the repository before allowing Servy to finalize the restart procedure.
To provide global oversight and detect catastrophic systemic failures—such as a corrupted configuration file pushed to the entire fleet or an extended API outage—the architecture integrates a remote Dead Man's Switch.19 While local nodes communicate their health via the heartbeat.json file in the centralized GitHub Gist, a total fleet failure would result in this file stagnating indefinitely. To monitor this, a GitHub Actions cron job will be deployed as a serverless watchdog.21 This workflow will execute on a high-frequency schedule, utilizing the Octokit API to poll the Gist. If the temporal delta between the current time and the last recorded heartbeat exceeds a predefined grace period, the workflow will trigger a global alert and lock the pending-merges.json state, freezing the fleet to prevent partially degraded nodes from executing rogue infrastructure mutations.23
Finally, to bound the risk associated with autonomous code modification, the ADR mandates the implementation of Algorithmic Circuit Breakers (ACBs).25 These stateful, external controls evaluate the agent's repair runs as they unfold, entirely independent of the LLM's output.25 If the self-repair cascade results in recursive crashing, the ACB will force a hard stop, preventing the LLM from destroying the local codebase or overwhelming the API quotas.25
The Metacognitive Log Review Framework
The deployment of autonomous agents within complex coding environments frequently exposes the limitations of standard ReAct (Reasoning and Acting) execution patterns. In a typical ReAct trace, an agent retrieves information, generates an internal thought, formulates an action, and observes the result.2 While this interleaving of logic theoretically enables self-correction, it often creates a dangerous illusion of competence.2 An agent may retrieve completely irrelevant documentation, fail to synthesize the correct API calls, and yet confidently generate plausible-sounding analysis, reporting total task completion with absolute conviction.2 The agent narrates whatever data it receives, entirely failing to notice the profound mismatch between its initial objective and its executed actions.2
This phenomenon, known as the coherence trap, is particularly destructive in automated code generation, where models can spend hours pursuing impossible solutions, burning through tens of thousands of tokens while generating increasingly incoherent logic.2 To counteract this, the dispatcher requires a persistent memory of its historical success rates and a mathematical framework to evaluate its current cognitive trajectory.
Calculating Semantic Drift Using Vector Embeddings
Evaluating the quality of every dispatcher execution log using an LLM judge is an anti-pattern. Continuous optimization requires extensive telemetry, but routing all implicit behavioral signals and explicit user logs back through an LLM incurs massive latency and exorbitant computational costs.26 Therefore, the metacognitive layer relies on a highly efficient, zero-cost pipeline utilizing semantic embeddings to detect behavioral anomalies, commonly referred to as data drift or concept drift.17
In the context of machine learning operations, concept drift occurs when the underlying relationship between inputs and correct outputs changes over time.27 Conversely, data drift refers to changes in the statistical properties of the input data itself.27 For the autonomous dispatcher, we are primarily concerned with embedding drift—a measurable shift in the semantic meaning of the agent's output over time.18
As the Node.js dispatcher processes tasks, it continuously appends detailed execution summaries, tool utilization rationales, and PR diff descriptions to the status/budget-dispatch-log.jsonl file. The metacognitive pipeline monitors this file, extracting the textual data representing the agent's "thought process."
To perform the mathematical analysis, the Node.js application utilizes a local, CPU-optimized embedding model. By integrating a library such as onnxruntime-node, the dispatcher can load a highly quantized, lightweight model (e.g., all-MiniLM-L6-v2). This model processes the textual logs and outputs dense numerical vectors representing the semantic essence of the agent's logic.17

Autonomous Self-Repair and Algorithmic Circuit Breakers
While the metacognitive layer ensures logical consistency, the physical execution environment requires an equally robust defense mechanism. When a fatal error occurs—such as a memory access violation or an unhandled exception thrown by a native module—the Node.js process crashes, triggering an Application Error within the Windows Event Viewer.4 At this critical juncture, the Servy Watcher daemon detects the unexpected termination and initiates the Autonomous Self-Repair Cascade via its Pre-Launch hook.11
The orchestrator of this cascade is a dedicated PowerShell script, repair_cascade.ps1. Its objective is to extract the diagnostic context, synthesize a repair using a secondary LLM, validate the code through strict syntax parsing, and apply the patch before allowing the system to restart.
The Repair Pipeline and Data Transformation Graphs
The first action of the repair orchestrator is to ingest the final lines of the dispatcher_stderr.log file, which Servy has carefully maintained and rotated.11 This log contains the specific stack trace, the exception signature, and the faulting module path.5
The script formulates a highly constrained prompt containing the stack trace and the source code of the broken module, transmitting this payload to a fresh Gemini or Mistral API instance. Crucially, the prompt engineering for this repair phase enforces a paradigm shift in how the LLM analyzes the code. Current automated program repair (APR) approaches typically employ a control-centric paradigm, forcing agents to navigate complex Code Property Graphs (CPGs) and irrelevant control logic.30 This often leads to semantic traps where the LLM patches the symptom rather than the underlying state defect.30
To resolve this, the prompt instructs the LLM to model the repair utilizing the principles of a Data Transformation Graph (DTG).30 By inverting the topology and modeling data states as nodes and functions as edges, the agent is forced to trace the logic defect through the actual data lineage rather than the superficial control flow.30 The LLM must return its proposed repair as a strictly formatted JSON object containing specific Abstract Syntax Tree (AST) mutation instructions, targeting the precise node where the data state becomes corrupted.
AST Validation and Sandboxed Execution
Blindly writing LLM-generated code directly to the file system is an unacceptable operational risk. Studies indicate that engineering teams frequently spend more time debugging AI-generated code vulnerabilities than they would writing the logic from scratch.31 AI coding assistants excel at superficial pattern matching but inherently lack an understanding of the hidden constraints within a specific project architecture.31
To mitigate this, the proposed patch is passed to a localized Node.js validation utility. This utility utilizes an AST parser, such as Acorn or Babel, to parse the broken source file into a traversable tree structure.32 The utility locates the targeted node and replaces it with the LLM's generated logic.32 The modified AST is then serialized back into JavaScript string format. If the LLM has generated syntactically invalid code, the serialization process will immediately throw a parsing error, failing the validation step and rejecting the patch.31
If the syntax is valid, the repaired module is injected into a pristine isolated-vm instance. The isolated-vm library provides a highly secure, lightweight V8 isolate that strictly sandboxes the execution environment, stripping the code of all access to the host file system, network APIs, and standard Node.js global objects [Context]. The validation utility executes the patched function against a suite of simulated data payloads, enforcing strict memory limits and synchronous execution timeouts. If the isolate executes without throwing an exception or exceeding the temporal bounds, the patch is formally validated. The orchestrator script utilizes native Git commands to create a hotfix branch, commit the AST-validated code, and push the patch to the central repository.33
Defining the Safety Boundaries: Algorithmic Circuit Breakers
The implementation of an autonomous self-repair loop introduces the severe risk of recursive failure. If the LLM hallucinates a syntactically valid but functionally destructive patch that simply moves the crash to a different line of code, the system will restart, crash again, and trigger another repair cycle. Left unconstrained, this loop will exhaust API budgets, spam the centralized GitHub repository with continuous hotfix branches, and permanently corrupt the system state.
To bound this risk and maintain a controlled blast radius, the architecture mandates the integration of Algorithmic Circuit Breakers (ACBs).25 ACBs are engineering patterns designed for hard-stop safety; they are stateful, external controls that can pause or halt an agent run based on measurable signals, operating entirely independent of the LLM's internal logic.25 They provide operational resilience by managing agent behavior when components fail, degrade, or produce unreliable outputs, ensuring graceful degradation rather than catastrophic looping.35
The repair cascade utilizes a three-state Operational Resilience Circuit Breaker to track the frequency of fatal crashes.35 The state is persistently recorded in a local circuit_state.json file.

	Closed

	Furthermore, the system employs specific Data Boundary Breakers during the repair phase.25 Before the PowerShell script transmits the raw stderr log to the LLM API, the Data Boundary Breaker scans the payload for sensitive information. It enforces trust zones by detecting and redacting API keys, environment variables, or Personally Identifiable Information (PII) that may have been inadvertently dumped into the stack trace, ensuring that sensitive internal data is never transmitted to external inference channels.25
Technical Implementation Plan
The deployment of the Metacognition and Self-Repair Layer requires a phased, rigorous implementation strategy across all decentralized host machines.
Phase 1: Deployment of the Out-of-Band Watcher
The foundation of the architecture requires the provisioning of the Servy daemon to replace all legacy process execution methods.
1. Installation and Initialization: Download the Servy executable and install it to the C:\Program Files\Servy directory on all participating Windows machines, ensuring the path is appended to the system environment variables.11
2. PowerShell Module Integration: Open an elevated PowerShell session and execute Import-Module "C:\Program Files\Servy\Servy.psm1" -Force to load the advanced management cmdlets.11
3. Service Splatting Configuration: Define the precise parameters for the dispatcher service using a PowerShell hashtable. This ensures that the health monitoring, log rotation, and critical Pre-Launch repair hooks are perfectly aligned.
PowerShell
$dispatcherConfig = @{
   Name = "AutonomousBudgetDispatcher"
   Path = "node"
   Params = "C:\FleetNode\dispatcher.mjs"
   Stdout = "C:\FleetNode\logs\dispatcher_stdout.log"
   Stderr = "C:\FleetNode\logs\dispatcher_stderr.log"
   EnableSizeRotation = $true
   RotationSize = 50
   MaxRotations = 5
   EnableHealth = $true
   RecoveryAction = "RestartProcess"
   PreLaunchPath = "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe"
   PreLaunchParams = "-File C:\FleetNode\scripts\repair_orchestrator.ps1"
   PreLaunchTimeout = 300
   PreLaunchIgnoreFailure = $false
}
Install-ServyService @dispatcherConfig

4. Verification: Execute Start-ServyService -Name "AutonomousBudgetDispatcher". Utilize the Get-ServyServiceStatus command to verify that the service transitions from StartPending to Running.11
Phase 2: Integration of the Metacognitive Drift Engine
With the process execution secured, the Node.js application must be augmented with the mathematical self-reflection capabilities.
   1. Dependency Provisioning: Within the Node.js project root, install the necessary numerical computation and local embedding libraries by executing npm install onnxruntime-node numeric.
   2. Model Loading: Implement an asynchronous initialization function to load a highly quantized version of the all-MiniLM-L6-v2 embedding model directly into the Node.js memory space, ensuring that vectorization occurs without network latency.
   3. Pipeline Construction: Create a background worker that utilizes the fs.watch API to monitor the status/budget-dispatch-log.jsonl file. As the primary agent writes new execution summaries and pull request descriptions, the worker extracts these strings, applies tokenization and lowercasing algorithms, and passes them to the local embedding model to generate the profile vector (  ).38
   4. Mathematical Evaluation: Implement the array processing logic to calculate the Exponential Moving Average (  ) and the cosine similarity (  ) against the historical baseline.16
   5. Intervention Routing: Integrate the output of the drift calculation directly into the primary execution loop of the dispatcher. If the calculated drift exceeds the    threshold, trigger an immediate state flush. The node must abort its current task, write a "Metacognitive Rejection" status to the centralized pending-merges.json Gist, and pause execution to prevent further degradation.18
Phase 3: Construction of the LLM Patch-Generation Loop
The core self-repair logic resides within the PowerShell orchestrator script invoked by Servy during a crash event.
      1. Circuit Breaker Initialization: The repair_orchestrator.ps1 script must first query the circuit_state.json file. It evaluates the timestamps of previous failures. If the failure count within the trailing 60-minute window exceeds the threshold (e.g., 3 failures), the script immediately exits with a non-zero status code.36 Servy, reading this failure code and respecting the PreLaunchIgnoreFailure = $false parameter, will abort the restart sequence, effectively halting the broken node.11
      2. Context Extraction and Redaction: If the circuit is Closed, the script utilizes Get-Content -Tail 150 to read the dispatcher_stderr.log.11 The script applies the Data Boundary Breaker, running regular expression sweeps to redact any API keys or secrets before packaging the stack trace.25
      3. LLM Orchestration: The script utilizes Invoke-RestMethod to transmit the redacted stack trace and the source code of the faulting module to the Gemini or Mistral API. The prompt mandates that the LLM analyzes the failure via Data Transformation Graphs (DTG) and returns a strict JSON payload containing the AST mutation requirements.30
      4. AST Validation and Sandboxing: The PowerShell script invokes a secondary Node.js utility, validate_patch.mjs. This utility utilizes the acorn parser to generate the AST, applies the LLM's mutations, and attempts to serialize the tree.32 If the syntax is valid, the patched code is injected into an isolated-vm environment and executed against mock data [Context].
      5. Git Integration: Upon successful isolated-vm validation, the PowerShell script executes native git checkout -b hotfix/auto-repair, commits the patched file, and utilizes the Octokit API to push the branch and open a Pull Request.33
      6. Service Restoration: The repair_orchestrator.ps1 script exits with code 0. Servy detects the successful completion of the Pre-Launch hook and proceeds to start the main Node.js process, bringing the node back online.11
Phase 4: Implementation of the Global Dead Man's Switch
To protect against systemic network failures or fleet-wide logical paralysis, a centralized watchdog must monitor the aggregate health of the decentralized nodes.
      1. Workflow Configuration: Create a new GitHub Actions workflow file located at .github/workflows/global-watchdog.yml within the central repository.
      2. Cron Scheduling: Configure the workflow to execute automatically on a high-frequency schedule using POSIX cron syntax: cron: '*/10 * * * *' (executing every ten minutes).22
      3. Gist Polling: Utilize the actions/github-script action to execute arbitrary JavaScript within the workflow. This script will utilize the injected Octokit client to perform a GET request against the centralized GitHub Gist containing the heartbeat.json payload.
      4. Temporal Analysis: The script parses the JSON payload to extract the UTC timestamp of the last recorded heartbeat from any node in the fleet. It calculates the differential between this timestamp and the current execution time.
      5. Alerting and Lockdown: If the differential exceeds the predefined 15-minute grace period, the workflow enters a critical failure state.23 It executes a webhook POST request to notify administrators via a designated communications channel. Simultaneously, the workflow utilizes Octokit to modify the centralized pending-merges.json file, applying a global lock flag. This ensures that even if a severely degraded node manages to regain partial connectivity, it is prohibited from merging corrupted code into the primary repository infrastructure.
